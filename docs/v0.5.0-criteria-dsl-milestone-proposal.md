# v0.5.0 Criteria DSL Milestone Proposal

Date: 2026-05-31

Status: Proposed

Proposed milestone name: `v0.5.0: Measurement-Backed Criteria DSL`

## Goal

Make criteria configuration easier to read, review, and validate by adding a measurement-backed TOML DSL that separates:

- the measurement to compute,
- the comparator to apply,
- and the required value with explicit units.

This milestone should turn the M6 criteria DSL direction into runtime behavior while preserving every existing `[[criteria]]` config shape.

## User Value

Electrical, test, and validation engineers should be able to read a TOML criterion and see the engineering requirement directly:

```toml
[[criteria]]
id = "rise_time_max_5ms"
channel = "switch_v"

[criteria.measurement]
type = "rise_time"
low_threshold = { value = 0.5, unit = "V" }
high_threshold = { value = 4.5, unit = "V" }

[criteria.requirement]
operator = "less_than_or_equal"
value = 0.005
unit = "s"
```

The existing explicit-field form remains supported:

```toml
[[criteria]]
id = "rise_time_max_5ms"
type = "rise_fall_time"
channel = "switch_v"
direction = "rise"
low_threshold_v = 0.5
high_threshold_v = 4.5
max_duration_s = 0.005
```

## Scope

- Add a TOML-deserializable DSL config model for measurement-backed criteria.
- Accept explicit units for initial measurement and requirement values.
- Support the initial operator vocabulary documented in `docs/criteria-dsl.md`.
- Map DSL criteria to the existing analysis and measurement engine behavior.
- Preserve existing `[[criteria]]` compatibility and report evidence fields.
- Add invalid-config validation tests with clear error messages.
- Add parity tests proving DSL configs produce the same JSON evidence as equivalent legacy configs.
- Add user-facing examples, migration notes, and schema/reference docs.

## Explicit Non-Goals

- No GUI.
- No DAQ integration.
- No plugin runtime.
- No batch analysis.
- No RTOS expansion.
- No hardware qualification or certification claim.
- No unit shorthand parser such as `"10ms"` in this milestone.
- No new measurement primitives such as duty cycle, RMS, frequency, overshoot, undershoot, peak-to-peak, or noise floor.
- No implicit unit conversion beyond explicitly supported units.
- No removal or reinterpretation of existing criteria fields.

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-036 | The config model shall support measurement-backed DSL criteria. | TOML supports `[criteria.measurement]` and `[criteria.requirement]` sections for existing measurement types without breaking existing criteria configs. |
| WRA-RQ-037 | The DSL shall use a small auditable operator vocabulary. | `less_than`, `less_than_or_equal`, `greater_than`, `greater_than_or_equal`, and `equal_to` are parsed, validated, tested, and reported. |
| WRA-RQ-038 | The DSL shall require explicit units. | Requirement and threshold values include explicit units; missing, unsupported, or mismatched units return clear errors without panics. |
| WRA-RQ-039 | DSL criteria shall evaluate through existing measurement evidence paths. | DSL and legacy criteria produce equivalent pass/fail decisions, measurement records, result fields, and JSON report evidence for equivalent configs. |
| WRA-RQ-040 | Existing criteria configs shall remain compatible. | Current fixtures, examples, and golden reports continue to pass without modification unless an intentional additive report field is approved. |
| WRA-RQ-041 | DSL invalid-config coverage shall be explicit. | Tests cover missing measurement sections, unknown operators, unsupported units, missing requirement values, incompatible measurement parameters, and duplicate/ambiguous criteria shapes. |
| WRA-RQ-042 | DSL docs shall be usable by engineering reviewers. | README, example config, criteria DSL docs, and report/schema docs show before/after configs, expected output, unit rules, and non-goals. |

## Architecture Direction

Recommended approach:

1. Keep the current `Criterion` evaluation engine as the behavioral baseline.
2. Add a config-layer representation for DSL criteria in `crates/wra-core/src/config.rs`.
3. Convert DSL criteria into the existing internal criteria model before analysis.
4. Reuse `wra-measurements` through `wra-core/src/analysis.rs`.
5. Preserve report measurement records and `measurement_id` links from M6.

This keeps the first DSL implementation mostly at the config boundary instead of forking the evaluator.

- Document that validation remains software known-answer validation, not hardware qualification or certification evidence.

## Risk Review

| Risk | Mitigation |
|---|---|
| DSL syntax may become ambiguous. | Keep DSL and legacy shapes mutually exclusive per criterion; reject mixed ambiguous fields. |
| Unit handling may imply unsupported conversion. | Require explicit supported units and reject unsupported units before evaluation. |
| Report consumers may see changed evidence shape. | Reuse M6 report schema and preserve existing result fields. |
| Existing user configs may break. | Run existing fixtures/golden tests unchanged and add compatibility tests. |
| Scope may drift into a general expression language. | Do not add arithmetic expressions, plugins, shorthand unit strings, or new measurements in v0.5.0. |
