# v0.7.0 Controller Simulation And Deployment Config System Milestone Proposal

Date: 2026-05-31

Status: Planned and merged through PR #87

Milestone: #9, `v0.7.0: Controller Simulation and Deployment Config System`

## Goal

Define the controller-in-the-loop layer that turns WRA into a qualification-to-production controller pipeline.

The intended workflow is:

```text
Desktop Simulator
  -> author production control config
  -> author test verification config
  -> observe UUT signals through fixtures or DAQ abstraction
  -> simulate controller behavior before final controller hardware exists
  -> generate qualification evidence
  -> export deployment package

RTOS / Controller Runtime
  -> load approved deployment package
  -> run production mode
  -> run test or signal-validation mode
  -> use shared configs, timing assumptions, and rule semantics
```

## Naming Note

The user-proposed direction used `M6` and `v0.6.0`. This repository already uses M6 for the measurement/evidence engine and `v0.6.0` for the planned portable rule package system. To preserve traceability, this future controller simulation milestone is named `M9 / v0.7.0`.

## User Value

Engineers can validate subcomponents before the final controller exists, tune controller configuration in a desktop environment, generate evidence, and later deploy approved configs into controller/runtime environments without reinterpreting the rules.

## Architecture Direction

The architecture anchor is `docs/controller-in-the-loop-workflow.md`.

Core concepts:

- a digital twin of the controller logic before physical controller completion
- separate but linked production control and test verification configs
- desktop simulation and RTOS runtime consuming shared definitions
- production mode and test/verification mode separation
- parity tests proving config and behavior consistency

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-051 | The project shall define a production control config schema. | Schema captures inputs, outputs, thresholds, state machine definitions, timing rules, control actions, fault responses, modes, version, and approval metadata without depending on desktop-only DAQ, plotting, or report code. |
| WRA-RQ-052 | The project shall define a test verification config schema separate from production control config. | Verification schema captures expected transitions, voltage limits, pulse widths, transient limits, dropout limits, stable-state requirements, timing windows, evidence/report settings, version, and approval metadata. |
| WRA-RQ-053 | The project shall add a virtual controller simulation engine. | Desktop simulation can run state-machine/controller logic against fixture or abstract sample input and produce control-state trace evidence without requiring final controller hardware. |
| WRA-RQ-054 | The project shall define a DAQ input abstraction. | DAQ-facing interfaces are abstracted with fixture/test-double support first; vendor SDKs, drivers, and live hardware remain behind separate dependency and environment gates. |
| WRA-RQ-055 | The project shall define a controller I/O abstraction. | Controller input/output interfaces separate portable controller logic from HAL, RTOS SDK, and hardware bindings. |
| WRA-RQ-056 | The project shall add a desktop simulation workflow. | CLI or future desktop workflow can load production control config, test verification config, channel map, and fixture/abstract input to produce simulation and verification evidence. |
| WRA-RQ-057 | The project shall define an RTOS deployment package format. | Deployment package links production control config, test verification config, channel map, manifest, checksum, qualification report, evidence SVG, and generated timestamp. |
| WRA-RQ-058 | The project shall separate production, test, and signal-validation modes. | Runtime mode definitions prevent production control behavior from being confused with test verification or signal-validation behavior. |
| WRA-RQ-059 | The project shall add config and behavior parity tests. | Same configs and waveform inputs produce matching desktop and embedded-compatible state, pass/fail, timing, channel, and evidence outputs, or document approved schema differences. |
| WRA-RQ-060 | The project shall define a qualification evidence report format for controller-in-the-loop workflows. | Reports link config versions, channel map, simulation trace, criteria evidence, deployment package metadata, checksum evidence, and explicit non-certification scope notes. |

## Validation Plan

Use representative fixture-driven scenarios before any live DAQ or hardware integration:

- switch signal with clean transition
- contact bounce/transient event
- dropout event
- slow rise/fall signal
- multi-channel input and output mapping
- controller state-machine transition trace

Validation remains software validation evidence only. Hardware qualification, production controller readiness, and certification claims require separate gates.

## Risk Review

| Risk | Mitigation |
|---|---|
| Production control and test verification configs may be conflated. | Keep separate schemas and link them through a manifest. |
| Desktop simulation may drift from RTOS runtime behavior. | Share configs, state-machine definitions, timing assumptions, and parity tests. |
| DAQ abstraction may pull in vendor SDK or hardware assumptions too early. | Start with interfaces and test doubles only; require dependency/environment gates for SDKs. |
| Controller I/O abstraction may imply hardware readiness. | Keep HAL/RTOS bindings out of scope until approved. |
| Evidence reports may be mistaken for certification evidence. | Include explicit non-certification scope notes and role review. |
