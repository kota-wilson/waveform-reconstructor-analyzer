# v0.11.0 Transform Runtime Profile Validation Milestone Proposal

Date: 2026-06-01

Status: GitHub milestone #13 and issues #158 through #163 created after M12 closure and explicit user approval to continue.

Milestone: #13, `v0.11.0: Transform Runtime Profile Validation`

## Goal

Add code-level runtime-profile validation for transform metadata so desktop-only transforms are not accidentally exposed as embedded, Pico, deployment, hardware, or certification support.

This milestone closes the documented M10/M12 gap without adding new transform algorithms, new dependencies, live DAQ support, HAL/RTOS SDK bindings, target hardware execution, GUI scope, binary package signing, or certification claims.

## Scope

| Area | Included In M13 | Excluded From M13 |
|---|---|---|
| Runtime profile validation | Structured validation API and tests for `desktop`, `pi5_no_std_candidate`, `pico2_candidate`, and `future_gated` transform profile requests. | Hardware boot evidence, RTOS runtime loading, HAL/SDK integration, or real-time claims. |
| Transform metadata | Validation over existing waveform, event, and validation transform metadata. | New FFT, spectral, resampling, calibration, high-pass baseline correction, or bounded embedded event runtime algorithms. |
| Timing evidence | Checks for sample-rate-required transforms when timing metadata is missing or invalid. | Clock-drift correction, resampling, jitter repair, or live DAQ timing validation. |
| Package guardrails | Documentation and tests that preserve current legacy rule-package behavior while making future transform-package exposure call the validator first. | Incompatible rule-package schema migration or binary deployment format changes. |

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-087 | FerrisOxide shall provide a structured transform runtime-profile validator. | A core API validates transform metadata against a requested runtime profile and returns stable error kinds, fields, transform names, requested profile, supported profiles, and reasons. |
| WRA-RQ-088 | Current desktop-only waveform transforms shall reject embedded, Pico, and future-gated runtime exposure. | Tests prove existing `[[filters]]` transform metadata passes for `desktop` and rejects `pi5_no_std_candidate`, `pico2_candidate`, and `future_gated` requests with clear errors. |
| WRA-RQ-089 | Sample-rate-required transforms shall require valid timing evidence. | Tests prove a sample-rate-required transform rejects missing, non-second, zero/negative, or non-finite timing metadata before profile exposure. |
| WRA-RQ-090 | Event and validation transform metadata shall remain desktop-only until a bounded runtime exists. | Tests prove M12 event/validation transform metadata passes for `desktop` and rejects embedded/Pico profile requests without claiming Schmitt event-report runtime support. |
| WRA-RQ-091 | Rule-package and deployment docs shall separate legacy export support from transform runtime support. | README, runtime-profile docs, and mapping docs state that current rule-package filter export is not a blanket transform runtime claim and future transform/package exposure must call the validator. |
| WRA-RQ-092 | M13 validation shall preserve dependency, schema, and hardware scope boundaries. | Workspace validation passes with no new third-party crates, no incompatible report/config schema changes, no live DAQ, no HAL/RTOS SDK, no target hardware execution, and no certification claims. |

## Verification Plan

- Unit tests for successful `desktop` validation.
- Unit tests for rejected `pi5_no_std_candidate`, `pico2_candidate`, and `future_gated` profile requests.
- Unit tests for missing and invalid sample timing on sample-rate-required metadata.
- Event/validation metadata tests using M12 transform metadata.
- Documentation review for package/export guardrail language.
- Workspace validation:
  - `cargo fmt --check`
  - `cargo test --workspace`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `git diff --check`

## Validation Plan

Validation remains software-only. The milestone validates that FerrisOxide rejects unsupported transform/runtime exposure before future runtime claims are made. It does not validate any hardware, DAQ, controller, RTOS, real-time, or certification behavior.

## Non-Goals

- No new signal-processing algorithms.
- No high-pass baseline correction implementation.
- No FFT, PSD, STFT, wavelet, resampling, calibration, or sensor-specific transforms.
- No new third-party dependencies.
- No live DAQ, vendor SDK, HAL, RTOS SDK, unsafe FFI, target hardware, or global setup.
- No binary deployment package serialization, signing, authentication, or tamper-proof claims.
- No hardware qualification, flight certification, or regulatory compliance evidence.

## Risk Review

| Risk | Mitigation |
|---|---|
| Runtime-profile validation may be mistaken for embedded runtime support. | Keep validator evidence scoped to metadata rejection and document that no hardware/runtime execution is added. |
| Existing rule-package filter export may be mistaken for broad transform runtime support. | Preserve legacy export wording while requiring future transform/package exposure to call the validator. |
| Timing validation may overclaim DAQ timing correctness. | Limit timing checks to metadata sufficiency; leave jitter, drift, resampling, and hardware clock behavior out of scope. |
| Event Schmitt no_std primitive may be mistaken for event-report runtime support. | Document that M12 report-oriented event pipeline remains desktop-only until a bounded runtime is separately approved. |
