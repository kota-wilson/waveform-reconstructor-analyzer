# v0.10.0 Event And Validation Transform MVP Milestone Proposal

Date: 2026-06-01

Status: Complete. PR #156 merged after required `rust` CI passed; issues #149 through #155 and milestone #12 are closed.

Milestone: M12 / #12, `v0.10.0: Event And Validation Transform MVP`

## Goal

Add event-oriented transforms and validation transforms that support switch, relay, actuator, and controller-test workflows.

The milestone should treat event extraction and pass/fail validation as separate layers. Event transforms produce auditable event records. Validation transforms evaluate requirements against those event records.

## Candidate Transform Set

| Transform | Category | Purpose |
|---|---|---|
| Dual threshold / Schmitt trigger | EventTransform / StatefulTransform | Convert analog samples into stable state with separate on/off thresholds. |
| Debounce | EventTransform | Reject short state changes around switch transitions. |
| Glitch removal | EventTransform | Remove pulses shorter than a configured duration. |
| Edge extraction | EventTransform | Produce rising/falling edge records. |
| Bounce detection | EventTransform / FeatureTransform | Count and characterize repeated transitions after actuation. |
| Missing pulse detection | ValidationTransform | Fail when an expected event does not occur. |
| Extra pulse detection | ValidationTransform | Fail when unexpected events occur. |
| Dwell-time validation | ValidationTransform | Require a state to persist for a duration. |
| Timeout validation | ValidationTransform | Fail if an event does not occur within a timing window. |

## Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-081 | Event transforms shall produce auditable event records separate from pass/fail validation. | Event records include transform name, channel, sample index, timestamp, state, thresholds, duration where applicable, and source transform metadata. |
| WRA-RQ-082 | Dual-threshold/Schmitt trigger shall convert analog samples to states with explicit thresholds and units. | Config validates on/off thresholds, units, channel, initial state, and hysteresis direction; tests cover chatter prevention and invalid thresholds. |
| WRA-RQ-083 | Debounce and glitch removal shall reject short pulses with documented time-window behavior. | Config validates finite non-negative durations; tests cover accepted, rejected, edge, and boundary-duration pulses. |
| WRA-RQ-084 | Edge and bounce detection shall report timestamps, sample indices, counts, and channel evidence. | Event fixtures produce known rising/falling edges, bounce counts, first/last bounce timestamps, and optional linked source events. |
| WRA-RQ-085 | Missing/extra pulse, dwell-time, and timeout validation shall produce pass/fail evidence linked to event records. | Validation results include requirement ID, measured value, required value, unit, channel, linked event IDs, and reason. |
| WRA-RQ-086 | Event transforms shall include known-answer switch/bounce fixtures and embedded-compatible parity where practical. | Known-answer fixtures document expected events before execution, and supported deterministic event logic has desktop-vs-embedded-compatible parity tests or documented non-parity reasons. |

## Verification Plan

- Unit tests for event record schema and validation result linking.
- Known-answer tests for Schmitt trigger output, debounce, glitch removal, edges, bounce count, dwell-time, timeout, missing pulse, and extra pulse behavior.
- Invalid-config tests for thresholds, units, durations, missing channels, and impossible timing windows.
- CLI JSON tests for event records and linked validation evidence.
- Parity tests for deterministic event logic that is suitable for embedded-compatible execution.
- Workspace validation:
  - `cargo fmt --check`
  - `cargo test --workspace`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `git diff --check`

## Validation Plan

Use representative signal cases:

- clean switch transition
- contact bounce after actuation
- short glitch that should be removed
- valid pulse train
- missing pulse
- extra pulse
- delayed response timeout
- heated actuator scenarios where response and transient checks can link to event records

Validation remains software-only and does not claim hardware qualification, live DAQ validation, controller timing guarantees, or certification evidence.

## Non-Goals

- No live DAQ, vendor SDK, driver, or hardware acquisition path.
- No HAL, RTOS SDK, unsafe FFI, or target hardware timing claim.
- No certification or hardware qualification claim.
- No spectral/time-frequency analysis.
- No advanced adaptive filtering or signal classification.
- No GUI or plugin runtime.

## Risk Review

| Risk | Mitigation |
|---|---|
| Event transforms may hide or reinterpret raw signal failures. | Preserve raw waveform data, keep derived event records explicit, and link validation decisions back to events. |
| Debounce/glitch windows may be unrealistic. | Require explicit durations, units, known-answer fixtures, and docs explaining tradeoffs. |
| Desktop and embedded event behavior may drift. | Use shared deterministic logic where practical and parity tests for embedded-compatible subsets. |
| Event records may be mistaken for hardware validation. | Keep software-only scope notes and require live hardware gates for DAQ or target execution. |
