# v0.8.0 Transform Architecture And Capability Metadata Milestone Proposal

Date: 2026-06-01

Status: Complete. PR #138 merged by squash commit `69b8b1a4a7c963316a74130655667ea3ff1481d5`; GitHub milestone #10 and issues #132 through #137 are closed.

Milestone: #10, `v0.8.0: Transform Architecture And Capability Metadata`

## Goal

Create the transform architecture needed to grow FerrisOxide beyond the current legacy `[[filters]]` config surface while preserving raw waveform data, existing behavior, and runtime boundaries.

The milestone should define metadata and validation before adding new transform algorithms. Existing moving average, low-pass, and ADC quantization behavior should be mapped into the new model without changing the current analysis results.

## Context

Current implemented transform behavior lives mostly in `crates/ferrisoxide-core/src/filter.rs` and is configured through `FilterConfig`. Waveform metadata records `transform_history` as strings. The user-provided taxonomy is broader than filters and includes pointwise transforms, baseline transforms, event transforms, calibration transforms, feature transforms, validation transforms, fault injection, quantization, and runtime constraints.

The architecture must support that future shape without implying every named transform is implemented.

## Proposed Architecture Direction

Define a transform capability model with at least these fields:

| Field | Purpose |
|---|---|
| `name` | Stable transform name exposed to config, docs, and reports. |
| `category` | One of the approved taxonomy categories. |
| `input_channels` | Channel requirements. |
| `output_channels` | Output channel behavior. |
| `parameters` | Typed parameter names, units, and validation rules. |
| `sample_rate_required` | Whether timing/sample-rate metadata is required. |
| `stateful` | Whether evaluation depends on previous samples or internal state. |
| `causal` | Whether transform can operate online without future samples. |
| `phase_effect` | `none`, `delay`, `nonlinear`, or `zero_phase`. |
| `streaming_supported` | Whether the transform can run over streaming input. |
| `offline_only` | Whether the transform requires full waveform access. |
| `runtime_profile` | Desktop, Pi 5 no_std candidate, Pico 2 candidate, or desktop-only. |
| `evidence_level` | Implemented, planned, research, or external-dependency-gated. |

Initial taxonomy categories should include:

- `PointwiseTransform`
- `WindowedTransform`
- `StatefulTransform`
- `FrequencyTransform`
- `TimeFrequencyTransform`
- `MultiChannelTransform`
- `CalibrationTransform`
- `EventTransform`
- `FeatureTransform`
- `ValidationTransform`

M10 should not require renaming existing `FilterConfig` immediately. It should define a compatibility path so existing configs and reports keep working while future configs can use the broader transform vocabulary.

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-070 | The project shall define a transform capability model and taxonomy vocabulary. | Docs and code design identify transform categories, metadata fields, implemented-vs-planned status, and runtime profile boundaries without claiming unimplemented transforms. |
| WRA-RQ-071 | Derived waveform metadata shall support structured transform metadata. | Reports can represent transform name, category, parameters, sample-rate requirement, statefulness, causality, phase effect, streaming support, offline-only status, runtime profile, and evidence level while preserving existing transform-history compatibility. |
| WRA-RQ-072 | Transform/runtime profile compatibility shall be validated before deployment exposure. | Desktop-only, Pi 5 no_std candidate, and Pico 2 candidate profiles reject unsupported transforms with clear errors before rule or deployment package exposure. |
| WRA-RQ-073 | Documentation shall distinguish implemented filters from planned transform families. | README, architecture docs, taxonomy docs, filter behavior docs, and ADC quantization docs make current support and future planning explicit, including non-goals and evidence limits. |
| WRA-RQ-074 | Existing transform behavior shall map to the new metadata without changing current analysis behavior. | Moving average, low-pass, and ADC quantization continue to pass existing tests and gain documented metadata mappings. |

## Verification Plan

- Existing workspace tests remain passing.
- Exact report tests prove additive metadata does not remove existing report fields.
- Config compatibility tests prove legacy `[[filters]]` entries still work.
- Unit tests verify metadata mappings for moving average, low-pass, and ADC quantization.
- Runtime-profile validation tests reject unsupported transform/profile combinations with structured errors.
- Documentation review confirms planned taxonomy entries are not described as implemented support.

## Validation Plan

Use existing representative fixtures:

- basic waveform
- noisy square wave
- analog switch bounce
- dropout event
- heated actuator passing and failing cases

Validation remains software validation evidence only. The milestone must not claim signal-processing accuracy beyond implemented equations and tests.

## Non-Goals

- No new signal transform algorithms.
- No new third-party crates.
- No live DAQ or hardware runtime integration.
- No HAL, RTOS SDK, unsafe FFI, or target hardware execution.
- No FFT, resampling, wavelet, sensor-calibration, or advanced filter-family implementation.
- No binary package serialization, signing, authentication, certification, or hardware qualification claims.

## Risk Review

| Risk | Mitigation |
|---|---|
| Taxonomy may be mistaken for implemented transform support. | Add evidence-level metadata and docs language that separates implemented, planned, and gated work. |
| Structured metadata may break existing report consumers. | Keep existing transform history additive, skip empty `transform_steps` for raw reports, preserve legacy filter config compatibility, and use golden tests for transformed reports. |
| Runtime capability claims may overstate embedded support. | Treat Pi 5 and Pico 2 support as capability profiles with validation gates, not hardware validation. |
| Transform profile validation may duplicate deployment package validation. | Keep transform capability validation focused on transform support and let deployment validation compose it later. |
