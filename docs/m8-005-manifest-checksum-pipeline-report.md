Owner Roles: Core Software Engineer / Security Engineer / Verification and Validation Engineer

## Objective

Add deterministic manifest and checksum evidence to exported FerrisOxide Rule Packages without adding signing, binary serialization, controller-runtime claims, or new dependencies.

## Scope Boundaries

In scope:

- `RulePackageManifest` schema types.
- Manifest source, validation, checksum, and artifact metadata.
- Deterministic `fnv1a64` artifact checksum helper.
- Structured checksum mismatch validation for artifact contents.
- CLI export of `manifest.json` and `checksum.txt`.
- Exact expected-artifact tests for manifest and checksum outputs.
- Dependency/security documentation for the non-cryptographic checksum choice.

Out of scope:

- Cryptographic signing, tamper-resistance claims, security certification, hardware qualification, or flight certification evidence.
- Binary package serialization.
- Runtime package loaders.
- Shared rule execution engine.
- no_std rule-engine boundary.
- Desktop-vs-embedded parity tests.
- GUI, DAQ, controller SDK, HAL, RTOS production integration, or hardware I/O.

## Acceptance Criteria Mapping

| Acceptance Criterion | Implementation |
|---|---|
| Package manifest records schema version. | `RulePackageManifest.schema_version` is copied from the exported package. |
| Package manifest records package version. | `RulePackageManifest.package.version` records package version. |
| Package manifest records target profile. | `RulePackageManifest.target.kind` and optional `identifier` record target profile data. |
| Package manifest records artifact list. | `RulePackageManifest.artifacts` records path, role, media type, checksum, and byte length. |
| Package manifest records source config. | `RulePackageManifest.sources.config` records the source TOML config path. |
| Package manifest records validation evidence links. | `RulePackageManifest.validation.report_artifact` links `validation-report.json`; `rule_package_validation` records `passed_before_export`. |
| Package manifest records checksum metadata. | `ChecksumMetadata` records algorithm, format, scope, and non-cryptographic security note. |
| Checksum output is deterministic. | `produces_deterministic_artifact_checksums` hard-codes a known `fnv1a64` output and CLI exact-artifact tests compare `checksum.txt`. |
| Mismatch tests return clear errors. | `validates_artifact_checksum_with_clear_mismatch_error` returns structured `ChecksumMismatch`. |
| Dependency/security review is completed before choosing checksum or binary-format dependencies. | `docs/dependency-review.md` records no new third-party dependency and explicitly excludes binary/signing/security claims. |

## Review Notes

- `fnv1a64` is deterministic and dependency-free, but it is intentionally not cryptographic.
- `checksum.txt` includes `rules.toml`, `rules.json`, `validation-report.json`, and `manifest.json`.
- `manifest.json` records checksums for the content artifacts and avoids self-referential checksum cycles.
- Export remains overwrite-safe through the existing create-new artifact writes.

## Retrospective

What worked:

- Keeping manifest and checksum data in `ferrisoxide-rule-schema` kept exported evidence aligned with package schema ownership.
- Exact artifact tests made deterministic checksum behavior visible in review.

What to watch:

- Future binary/signing work must not reuse the non-cryptographic checksum as a security guarantee.
- no_std boundary work may need feature-gating for serde/TOML/JSON helpers.
