# v0.6.0 Portable Rule Package System Milestone Proposal

Date: 2026-05-31

Status: Planned

Milestone: #8, `v0.6.0: Portable Rule Package System`

## Goal

Turn WRA into a desktop-to-embedded qualification pipeline by introducing a portable WRA Rule Package that can be authored and validated on the desktop, then consumed by an embedded/controller runtime without duplicating rule semantics.

The product direction is:

```text
Desktop WRA
  -> author criteria
  -> simulate filters
  -> validate against CSV/test data
  -> generate qualification evidence
  -> export WRA Rule Package
  -> RTOS / controller runtime consumes the same rule package
```

## Naming Note

The user-proposed direction used `M5` and `v0.5.0` wording. This repository already uses M5 for plotting and has an active `v0.5.0: Measurement-Backed Criteria DSL` milestone. To preserve traceability, this future package-system milestone is named `M8 / v0.6.0`.

## User Value

Electrical engineers, test engineers, embedded engineers, and verification engineers should be able to review one rule definition and know that:

- Desktop simulation used the same criteria semantics intended for deployment.
- Exported deployment artifacts include version, units, channel mapping, timing assumptions, and checksum evidence.
- Embedded/controller runtime behavior is checked against desktop behavior with parity tests.
- The desktop evidence package remains human-readable while the controller runtime consumes a constrained deployment subset.

## Architecture Direction

The controlling architecture decision is recorded in `decisions/ADR-004-portable-rule-package-architecture.md`.

Planned boundaries:

| Component | Responsibility |
|---|---|
| `wra-rule-schema` | Owns the versioned portable rule-package model. |
| `wra-rule-engine` | Executes validated rules through shared semantics. |
| `wra-cli` | Adds desktop package export commands. |
| `wra-embedded` | Provides no_std adapter boundaries for embedded-compatible evaluation. |
| `wra-controller-runtime` | Future RTOS/controller deployment adapter. |

The architecture should enforce:

```text
one rule schema
one shared rule engine
multiple frontends and runtimes
```

## Rule Package Fields

Initial package schema should include:

- package name
- package version
- target profile
- schema version
- channel mapping
- channel units
- sample-rate assumptions
- filter definitions
- criteria definitions
- thresholds
- timing limits
- explicit units
- validation metadata
- manifest metadata
- checksum metadata

## Deployment Package

The desktop export path should produce a human-reviewable package:

```text
deployment/
  rules.toml
  rules.json
  rules.bin
  manifest.json
  checksum.txt
  validation-report.json
  qualification-evidence.svg
```

Embedded/controller runtimes should start with a smaller consumption target:

```text
rules.bin
manifest.json
checksum.txt
```

## Scope

- Add a versioned portable rule package schema.
- Define the desktop export artifact set.
- Add package validation with clear structured errors.
- Add a package export command.
- Add manifest and checksum evidence.
- Introduce a shared rule execution engine boundary.
- Define the no_std compatibility boundary for embedded consumption.
- Add desktop-vs-embedded parity tests.

## Explicit Non-Goals

- No GUI.
- No live DAQ integration.
- No controller SDK or hardware HAL.
- No Zephyr production integration.
- No RTOS timing guarantees.
- No certification claim or hardware qualification claim.
- No encrypted, signed, or security-certified package format in the first slice.
- No unit shorthand strings such as `10ms`.
- No separate desktop-only and embedded-only rule semantics.
- No binary package format without dependency/security review and a deterministic test plan.

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-043 | The project shall define a versioned portable WRA Rule Package schema. | A schema model captures package metadata, channels, units, sample-rate assumptions, filters, criteria, thresholds, timing limits, and target profile without depending on CLI, CSV, plotting, or controller I/O. |
| WRA-RQ-044 | The project shall define a reviewable deployment package artifact set. | Documentation and tests define `rules.toml`, `rules.json`, future `rules.bin`, `manifest.json`, `checksum.txt`, validation report, and evidence SVG roles. |
| WRA-RQ-045 | The project shall validate rule packages before export or execution. | Invalid packages return structured errors for missing channels, unsupported units, unknown criteria, invalid timing/sample-rate assumptions, checksum mismatch, and incompatible target profiles. |
| WRA-RQ-046 | The desktop CLI shall export rule packages. | A CLI command exports the rule package artifacts from validated config and analysis evidence without changing existing analyze/plot behavior. |
| WRA-RQ-047 | Rule packages shall include manifest and checksum evidence. | Exported packages include deterministic manifest metadata and checksum files; checksum algorithm and dependencies pass security/dependency review before implementation. |
| WRA-RQ-048 | The project shall use one shared rule execution engine for desktop and embedded-compatible paths. | Desktop/CLI and embedded-compatible runtime tests execute through the same rule-engine semantics instead of duplicating criteria behavior. |
| WRA-RQ-049 | The embedded rule path shall preserve a no_std compatibility boundary. | Embedded-compatible rule execution avoids CSV parsing, file I/O, plotting, report generation, hardware HALs, and heap requirements for basic evaluation where practical. |
| WRA-RQ-050 | The project shall prove desktop-vs-embedded rule parity with exact tests. | The same waveform, same rule package, and same expected result are evaluated by desktop and embedded-compatible paths with exact matching pass/fail and evidence values. |

## Validation Plan

Use existing known-answer and environmental fixtures first:

- clean square wave
- dropout event
- contact bounce/transient fixture
- slow rise/fall signal
- measurement-engine known-answer fixture

Each parity case should include:

```text
tests/parity/
  waveform_001.csv
  rules_001.toml
  expected_result.json
```

Both desktop and embedded-compatible engines must produce the same result. This remains software validation evidence, not hardware qualification or certification evidence.

## Risk Review

| Risk | Mitigation |
|---|---|
| Desktop and controller behavior may drift. | Use one shared schema, one rule engine, and exact parity tests. |
| Deployment packages may be overread as certified controller releases. | Label package evidence as software validation output only. |
| Binary formats and checksums may introduce dependency/security risk. | Require dependency and security review before checksum/binary implementation. |
| Embedded path may absorb desktop concerns. | Keep CSV, plotting, reports, and file I/O out of no_std runtime crates. |
| Active DSL milestone may be disrupted. | Keep v0.6.0 as a future milestone after v0.5.0 unless explicitly reprioritized. |
