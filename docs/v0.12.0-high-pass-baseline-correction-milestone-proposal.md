# v0.12.0 High-Pass Baseline Correction Milestone Proposal

Date: 2026-06-01

Status: Complete through PR #173. GitHub milestone #14 and issues #167 through #172 are closed.

Milestone: #14, `v0.12.0: High-Pass Baseline Correction`

## Goal

Implement the deferred first-order high-pass baseline correction transform from WRA-RQ-078 as a desktop analysis transform with explicit timing, phase, metadata, and validation boundaries.

This milestone adds one causal software transform. It does not add a broad filter-design framework, new dependencies, live DAQ support, HAL/RTOS SDK bindings, target hardware execution, rule-package transform export, GUI scope, binary package signing, or certification claims.

## Scope

| Area | Included In M14 | Excluded From M14 |
|---|---|---|
| Transform config | Add `[[filters]] type = "high_pass_baseline"` with `cutoff_hz`. | New config table families, unit shorthand parsing, or incompatible config/schema migration. |
| Transform implementation | First-order causal high-pass baseline correction over each channel. | FIR/IIR design framework, Butterworth/Chebyshev/Elliptic families, zero-phase filtering, high-order filters, or calibration models. |
| Timing validation | Require finite positive cutoff and strictly increasing time samples. | Jitter correction, resampling, clock-drift correction, live DAQ timing validation, or target timing guarantees. |
| Evidence | Unit tests, CLI/config coverage, raw-preservation checks, metadata checks, docs, traceability, and risk updates. | Hardware validation, certification evidence, runtime-loader exposure, rule-package export support, or embedded profile support. |

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-093 | FerrisOxide shall support a `high_pass_baseline` filter config. | TOML config accepts `type = "high_pass_baseline"` with required finite positive `cutoff_hz`, converts it to a filter step, and rejects missing or invalid values with clear errors. |
| WRA-RQ-094 | FerrisOxide shall implement first-order high-pass baseline correction. | The transform uses a documented causal recurrence over strictly increasing time samples, removes constant baseline input, and preserves input channel names, units, and raw samples. |
| WRA-RQ-095 | High-pass baseline correction shall validate timing and sample assumptions. | The transform rejects zero/negative/non-finite cutoff, non-finite/duplicate/decreasing timestamps, and non-finite samples before producing output. |
| WRA-RQ-096 | High-pass baseline correction shall emit structured transform metadata. | Reports include `high_pass_baseline` transform metadata with `cutoff_hz` in `Hz`, `sample_rate_required = true`, `stateful = true`, `causal = true`, `phase_effect = delay`, `streaming_supported = true`, `offline_only = false`, and `runtime_profiles = ["desktop"]`. |
| WRA-RQ-097 | High-pass baseline correction shall have CLI/config and export guardrail coverage. | A config-driven CLI test proves analysis succeeds with `high_pass_baseline`, and rule-package export continues to reject unsupported transform export with a clear error. |
| WRA-RQ-098 | M14 shall document behavior and preserve scope boundaries. | Filter behavior, transform metadata, capability model, runtime-profile docs, requirements, traceability, risk, and validation logs state software-only desktop scope and no DAQ, HAL/RTOS, hardware, certification, or broad filter-family claims. |

- Unit tests for constant-baseline removal and synthetic slow-drift behavior.
- Unit tests for invalid `cutoff_hz`, non-finite/duplicate/decreasing timestamps, and non-finite samples.
- Metadata tests for transform history, structured transform step fields, and raw data preservation.
- Config conversion tests for `high_pass_baseline`.
- CLI JSON smoke test using an example config.
- Rule-package export rejection test for unsupported `high_pass_baseline` export.
- Workspace validation:
  - `cargo fmt --check`
  - `cargo test --workspace`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `git diff --check`

## Validation Plan

Validation remains software-only. M14 validates the implemented recurrence, timing preconditions, metadata, and CLI/config behavior against deterministic synthetic fixtures. It does not validate hardware acquisition, real-time operation, DAQ clock behavior, sensor calibration, or certification suitability.

## Non-Goals

- No new third-party dependencies.
- No broad FIR/IIR filter-design framework.
- No Butterworth, Chebyshev, Elliptic, Bessel, notch, band-pass, zero-phase, or high-order filter implementation.
- No rule-package or deployment-package transform export expansion.
- No embedded/no_std runtime exposure.
- No live DAQ, vendor SDK, HAL, RTOS SDK, unsafe FFI, target hardware, or global setup.
- No hardware qualification, flight certification, or regulatory compliance evidence.

## Risk Review

| Risk | Mitigation |
|---|---|
| High-pass filtering may hide real low-frequency failures. | Preserve raw data, require explicit config, document recurrence and phase behavior, and emit structured metadata. |
| Users may mistake software filtering for calibrated sensor correction. | Keep docs scoped to derived software transforms and avoid calibration/physical-truth wording. |
| Timing-dependent behavior may be misused on invalid time axes. | Reject duplicate/decreasing timestamps and invalid cutoff values before producing output. |
| Runtime-profile validation may be overread as embedded support. | Keep runtime profile desktop-only and rule-package export unsupported in M14. |
