# v0.9.0 Pointwise And Windowed Transform MVP Milestone Proposal

Date: 2026-06-01

Status: Proposed locally. No GitHub milestone or issues have been created yet.

Milestone: proposed M11, `v0.9.0: Pointwise And Windowed Transform MVP`

## Goal

Add the first post-architecture transform implementations after M10 establishes transform capability metadata and compatibility rules.

This milestone should focus on transforms that are useful in DAQ/test workflows, easy to verify with deterministic fixtures, and implementable without new third-party dependencies.

## Candidate Transform Set

| Transform | Category | Purpose |
|---|---|---|
| Offset / bias add | PointwiseTransform | Add or remove DC offset. |
| Gain / scale | PointwiseTransform | Convert or scale signal magnitude. |
| Inversion | PointwiseTransform | Flip polarity. |
| Clamp / limit | PointwiseTransform | Bound min/max values. |
| Deadband | PointwiseTransform | Ignore small changes around zero. |
| DC removal | Baseline / WindowedTransform | Remove mean value from a channel. |
| Baseline subtraction | Baseline / PointwiseTransform | Subtract a measured or configured idle baseline. |
| Moving median | WindowedTransform | Smooth spikes better than a moving average. |
| First-order high-pass baseline correction | StatefulTransform | Reduce low-frequency baseline wander when time-axis assumptions are valid. |

The first-order high-pass baseline correction can be split out if M11 becomes too large.

## Proposed Requirements

| ID | Requirement | Acceptance Criteria |
|---|---|---|
| WRA-RQ-075 | Pointwise transform config shall support offset, gain, inversion, clamp, and deadband. | Config parsing, validation, transform execution, report metadata, and invalid-parameter tests cover all five transforms without changing raw data. |
| WRA-RQ-076 | Baseline transforms shall support DC removal and baseline subtraction with explicit assumptions. | DC removal and baseline subtraction document units, channel scope, sample assumptions, and derived-waveform lineage; tests cover mean removal and configured baseline subtraction. |
| WRA-RQ-077 | Moving median shall be supported as a windowed transform with defined edge behavior. | Moving median validates window size, documents trailing or centered behavior, handles edges deterministically, and passes known-answer tests. |
| WRA-RQ-078 | First-order high-pass baseline correction shall document phase and edge behavior and require valid time-axis metadata. | The transform rejects invalid time axes, records phase effect and sample-rate requirement, and passes synthetic drift tests if included in M11. |
| WRA-RQ-079 | Raw data preservation and transform metadata shall be proven for every new transform. | Tests prove source waveform samples remain unchanged and derived waveform metadata records structured transform details and legacy transform history. |
| WRA-RQ-080 | Docs and examples shall show transforms as derived artifacts, not calibrated physical truth. | README, filter/transform docs, and examples avoid calibration, hardware accuracy, production readiness, or certification claims. |

## Verification Plan

- Unit tests for each transform's numeric behavior.
- Config tests for valid and invalid parameters.
- CLI tests applying transforms before criteria.
- Golden JSON report tests for transform metadata.
- Raw preservation tests for every transform.
- Edge-behavior tests for moving median and high-pass baseline correction if included.
- Workspace validation:
  - `cargo fmt --check`
  - `cargo test --workspace`
  - `cargo clippy --workspace --all-targets -- -D warnings`
  - `git diff --check`

## Validation Plan

Use synthetic and known-answer fixtures:

- clean square wave with offset/gain transforms
- noisy square wave with moving median
- slow baseline drift fixture
- analog switch bounce fixture with and without deadband
- heated actuator fixture to prove current criteria behavior still runs over derived waveforms

Validation remains software-only. These transforms do not prove sensor calibration accuracy or hardware performance.

## Non-Goals

- No FFT, PSD, STFT, wavelet, or frequency-domain transforms.
- No resampling, interpolation, or clock-drift correction.
- No sensor-specific calibration claims.
- No new third-party dependencies.
- No live DAQ, controller runtime, HAL, RTOS SDK, or target hardware work.
- No binary deployment package or signing work.

## Risk Review

| Risk | Mitigation |
|---|---|
| Pointwise transforms may be mistaken for calibrated engineering-unit conversion. | Document units and limitations; avoid calibration claims unless backed by later calibration-specific requirements. |
| Baseline removal may hide real failures. | Preserve raw data, record transform metadata, and make baseline transforms explicit in reports. |
| Moving median edge behavior can change expected timing evidence. | Define edge behavior before implementation and cover known-answer fixtures. |
| High-pass baseline correction may introduce phase delay or reject valid non-uniform timing. | Require time-axis validation and record phase effect. |
