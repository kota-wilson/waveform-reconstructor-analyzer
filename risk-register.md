| WRA-RISK-016 | Annotated SVG evidence may be overread as complete engineering proof instead of visual software evidence. | Medium | High | Reuse report measurement evidence, label scope in plotting docs, keep overlays 2D/SVG-only, and avoid hardware or certification claims. | Documentation Engineer / Verification and Validation Engineer | Active | Plot evidence overlay changes. |

| WRA-RISK-019 | Desktop and embedded/controller rule behavior may drift if rule semantics are duplicated. | Medium | High | Use one portable rule schema, one shared rule engine, and exact desktop-vs-embedded parity tests before runtime claims. M8-001 creates the schema boundary; M8-003 adds pre-export/pre-execution validation; execution, no_std, and parity evidence remain separate gates. | Software Architect / Verification and Validation Engineer | Active | Portable rule package or controller-runtime work. |
| WRA-RISK-020 | Deployment packages may be mistaken for certified controller releases or hardware qualification evidence. | Medium | High | Label package outputs as software validation evidence only; avoid controller production-readiness, real-time safety, and certification claims until separate gates exist. | WRA-RISK-021 | Manifest, checksum, or binary package work may create security/dependency overclaims. | Medium | Medium | Require dependency and security review before checksum algorithms, binary serialization, signing, or package integrity claims; test deterministic outputs and mismatch errors. | WRA-RISK-022 | Production control config and test verification config may be conflated. | Medium | High | Keep separate schemas and link them through deployment manifests, version metadata, and parity tests. | Software Architect / Verification and Validation Engineer | Active | Controller-in-the-loop config work. |
| WRA-RISK-023 | DAQ or controller I/O abstractions may pull in vendor SDK, HAL, or hardware assumptions too early. | Medium | High | Start with abstractions and test doubles only; require dependency, environment, security, and hardware gates before SDK/HAL integration. | Systems Engineer / Security Engineer | Active | DAQ, controller I/O, or RTOS runtime changes. |
| WRA-RISK-024 | Desktop virtual controller behavior may drift from RTOS runtime behavior. | Medium | High | Share state-machine definitions, timing assumptions, config schemas, and exact desktop-vs-embedded parity tests. | Embedded RTOS Engineer / Core Software Engineer | Active | Simulator or runtime implementation. |

## Escalation

Critical impact or safety/security risks must be escalated to the Technical Director before release.
